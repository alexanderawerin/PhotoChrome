# PhotoChrome Architecture and Code Audit

Date: 2026-06-28

## Executive Summary

PhotoChrome has a sound product foundation: media stays in the browser, recipe data is runtime-validated, expensive work is increasingly moved off the main thread, and the main photo workflows have useful E2E coverage. The largest technical risk is not the choice of React, Canvas, or WebGL; it is behavioral drift between the CPU, worker, and WebGL implementations of the same recipe pipeline.

The audit fixed confirmed drift in Kelvin export and video preprocessing, made LUT loading deterministic, removed a Smart Picks listener leak and stale preview race, normalized JPEG filenames, cleared ESLint warnings, and updated vulnerable dependencies. Production dependencies now report zero known vulnerabilities.

The recommended next product feature is **Batch Export**. Multi-image upload and "Apply to all" already create most of a batch workflow, but users can only export the current image. Completing that loop offers more user value than adding another discovery feature immediately after Smart Picks.

## Architecture

### Strengths

- `src/engine` is largely framework-independent and uses typed processing contracts.
- Recipe and simulation JSON is validated at module load.
- Photo export uses a worker, while video processing uses a dedicated WebGL instance.
- LUT loads are cached and deduplicated.
- CPU fallback keeps photo processing available when WebGL is unavailable.
- The application is local-only by design; no media upload path was found.
- CI covers lint, unit tests, production build, and Chromium E2E.
- All 100 recipe IDs are unique and reference known simulations.

### Pressure Points

- `engine/video.ts` (861 lines), `engine/webgl/processor.ts` (783), `Editor.tsx` (currently about 750), and `RecipePanel.tsx` (611) carry several responsibilities each.
- Recipe behavior is implemented in CPU, worker, and shader code. Without parity tests, new settings can be added to one path and omitted from another.
- Full-resolution `ImageData` objects are held in React state. Multi-image loading has no count, byte, or pixel budget and decodes original plus thumbnail concurrently.
- Global singleton processors and module-level preview caches have bounded entries but no lifecycle tied to an editing session.
- Video has no automated browser coverage despite having the most browser-dependent code.

## Confirmed Fixes

1. **Kelvin missing from full-resolution photo export**
   - Cause: `processor.worker.ts` duplicated preprocessing but omitted `whiteBalanceKelvin`.
   - Fix: shared `applyPreprocessSettings` is now used by sync and worker paths, with unit coverage.

2. **Cold-cache LUT previews were order-dependent**
   - Cause: recipe cards and video processing read `getCachedLUT` without owning the preceding load.
   - Fix: cards explicitly await LUT loading; video waits for its simulation before producing processing options.

3. **Video ignored Dynamic Range, WB preset, and Kelvin**
   - Cause: WebGL uniforms only represented WB shift and post-simulation settings.
   - Fix: shader preprocessing now mirrors CPU order and coefficients before LUT/curve application.

4. **Stale photo preview race**
   - Cause: asynchronous LUT loads could update the shared preview after the user switched image or recipe.
   - Fix: preview work is centralized in a cancellable effect.

5. **Smart Picks listener accumulation**
   - Cause: one message listener was added per request and stale listeners were never removed when another request completed first.
   - Fix: one worker listener now routes responses through a pending-request map; the cache now refreshes entries on access.

6. **JPEG downloaded with source extension**
   - Cause: export always encoded JPEG but reused `.png`/`.webp` from the input filename.
   - Fix: exported photo names always end in `.jpg`.

7. **Lint and dependency baseline**
   - ESLint now completes with zero warnings.
   - Co-located CVA variants in design-system modules are explicitly classified as an intentional Fast Refresh exception.
   - PostCSS, Vite, Babel, and js-yaml were updated within compatible ranges; `npm audit` reports zero vulnerabilities.

## Recommended Product Step: Batch Export

### Why now

- Multi-image upload, navigation, per-image settings, and "Apply to all" already exist.
- Exporting only the current image makes the multi-image workflow stop before its natural completion.
- Smart Picks improved recipe discovery; Batch Export converts that discovery into a complete high-frequency workflow.
- The feature reinforces the privacy-first position because the entire batch can remain local.

### Minimum useful scope

1. Export every image that has an active recipe.
2. Process sequentially or with very small bounded concurrency.
3. Show `current / total`, current filename, and a cancel action.
4. Produce consistently named JPEG files and preserve recipe EXIF metadata.
5. Package results into one archive or use a browser-supported directory workflow with a clear fallback.
6. Continue after an individual failure and provide a final success/failure summary.

### Required foundation

Implement media budgets and worker cancellation first or in the same milestone. Batch processing on the current unbounded memory model would amplify the primary reliability risk.
